# 工作流 09：从混乱需求到可执行项目计划的 AI 多专家工作流

**适用岗位**：项目经理、产品经理、团队负责人、技术负责人

**真实场景**：你收到一封邮件，或者开完一个需求会议，手里攥着一堆零散的需求描述、口头承诺和模糊目标，领导要求三天内给出项目计划。

**最终目标**：产出一套可执行的项目计划，包括需求清单、任务拆解、风险清单、排期计划和沟通计划。

---

## 输入材料清单

开始前请准备以下材料：

- 原始需求文本（邮件正文、会议记录截图文字、口头说明整理稿）
- 项目截止日期或期望上线时间
- 可用的人力资源信息（几个人、各自擅长什么）
- 已知的约束条件（预算上限、技术限制、外部依赖）

---

## 工作流总览

```
节点 1：需求澄清专家
  ↓ 输出：结构化需求陈述清单
节点 2：干系人识别专家
  ↓ 输出：干系人清单（角色、关注点、影响程度）
节点 3：任务拆解专家
  ↓ 输出：任务清单（含依赖关系）
节点 4：风险识别专家
  ↓ 输出：风险清单（含应对措施）
节点 5：排期规划专家
  ↓ 输出：项目排期表
节点 6：资源评估专家
  ↓ 输出：资源需求报告
节点 7：沟通计划专家
  ↓ 输出：项目沟通计划
节点 8：项目复盘专家
  ↓ 输出：复盘框架模板
最终交付：完整项目计划文档
```

---

## 专家节点详解

---

### 节点 1：需求澄清专家

**职责**：将原始混乱需求整理为结构清晰的需求陈述，识别模糊点和缺失信息。

**输入**：原始需求文本（邮件、会议记录、口头说明）

**输出**：结构化需求陈述清单（含背景、目标、功能需求、非功能需求、待确认项）

#### 基础提示词

```
请将以下混乱的需求描述整理为结构化的需求陈述。

原始需求：【粘贴原始需求文本】

请输出：
1. 项目背景（一句话）
2. 核心目标（不超过 3 条）
3. 功能需求清单（每条一句话，编号）
4. 非功能需求（性能、安全、兼容性等）
5. 待确认项（模糊或缺失信息的列表，标注"需要向谁确认"）
```

#### 进阶提示词

```
你是一位资深产品经理，专注于需求分析和问题澄清。

请对以下原始需求进行结构化分析，并识别所有需要确认的模糊点。

原始需求：【粘贴原始需求文本】

分析维度：
1. 区分"必须有"、"应该有"、"可以有"三类需求（MoSCoW 法）
2. 对每条功能需求，判断是否有明确的验收标准
3. 识别需求之间的矛盾或冲突
4. 列出所有假设前提（项目团队未经确认就默认成立的条件）
5. 给出 5 个关键澄清问题，排序按优先级

输出格式：需求分类表 + 假设前提清单 + 澄清问题清单
```

#### 审查提示词

```
请检查以下需求陈述是否存在以下问题：
1. 是否有需求没有明确的验收标准？
2. 是否有"必须有"和"可以有"被混淆的需求？
3. 待确认项是否都指明了需要向谁确认？
4. 是否有两条需求实际上描述的是同一件事？

【粘贴需求陈述清单】

逐项说明问题。
```

#### 修正提示词

```
根据审查意见，修改需求陈述清单。为没有验收标准的需求补充验收标准，澄清需求分类，合并重复需求。

原始清单：【粘贴原始需求清单】
审查意见：【粘贴审查结果】
```

**验收标准**：

- [ ] 每条功能需求是否都有明确的验收标准？
- [ ] 待确认项是否每条都注明了"需向谁确认"？
- [ ] 是否区分了必须有和可以有的需求？
- [ ] 需求之间是否没有明显矛盾？

---

### 节点 2：干系人识别专家

**职责**：识别项目所有相关干系人，明确他们的关注点和对项目的影响程度。

**输入**：节点 1 的结构化需求清单

**输出**：干系人清单（含姓名/角色、关注点、影响程度、沟通策略建议）

#### 基础提示词

```
请根据以下项目需求，识别所有可能的干系人。

项目需求摘要：【粘贴需求清单中的核心目标和功能需求】

对每个干系人输出：
- 角色（例如：发起人、用户代表、技术负责人、外部供应商）
- 主要关注点（他们最在意什么）
- 对项目的影响程度（高/中/低）
- 参与方式（决策者/执行者/知情者/审批者）
```

#### 进阶提示词

```
你是一位资深项目经理，擅长干系人管理。请根据以下项目信息，完成干系人分析，并给出管理策略。

项目概要：【粘贴项目背景和核心目标】
组织背景：【填入组织类型，例如：互联网公司/政府单位/制造业】

请完成：
1. 干系人识别（包含隐性干系人，例如：会被项目影响但没有主动参与的部门）
2. 干系人权力/利益矩阵（高权力高利益 / 高权力低利益 / 低权力高利益 / 低权力低利益）
3. 每类干系人的管理策略（密切合作 / 保持满意 / 随时告知 / 监控即可）
4. 潜在干系人冲突识别（哪两个干系人的利益可能冲突）

输出格式：表格 + 冲突分析段落
```

#### 审查提示词

```
请检查以下干系人清单是否存在以下问题：
1. 是否遗漏了会被项目影响但未被列入的人或部门？
2. 是否有干系人的关注点描述过于模糊（例如只写"关注项目进度"）？
3. 影响程度评估是否有明显不合理的地方？

【粘贴干系人清单】

逐项说明问题。
```

#### 修正提示词

```
根据审查意见，补充遗漏的干系人，修正过于模糊的关注点描述，调整不合理的影响程度评估。

原始清单：【粘贴原始干系人清单】
审查意见：【粘贴审查结果】
```

**验收标准**：

- [ ] 是否包含了所有会影响项目或被项目影响的关键角色？
- [ ] 每个干系人的关注点是否具体（而不是"关注项目成功"这类废话）？
- [ ] 是否标注了每人的参与方式（决策者/知情者）？

---

### 节点 3：任务拆解专家

**职责**：将功能需求拆解为可执行的具体任务，明确每个任务的负责人类型和任务间的依赖关系。

**输入**：节点 1 的需求清单 + 节点 2 的干系人清单

**输出**：任务清单（含任务名、负责人角色、预估工作量、依赖关系、交付物）

#### 基础提示词

```
请将以下功能需求拆解为具体可执行的任务清单。

功能需求：【粘贴功能需求列表】

对每个任务输出：
- 任务编号（T-001 格式）
- 任务名称（动词开头，20 字以内）
- 负责人角色（前端/后端/产品/设计/测试/运营）
- 预估工作量（人天）
- 前置依赖（哪个任务完成后才能开始）
- 交付物（任务完成的标志是什么）
```

#### 进阶提示词

```
你是一位资深项目经理，专注于工作分解结构（WBS）设计。

请对以下需求进行三级任务分解：
- 第一级：功能模块（例如：用户系统、支付模块）
- 第二级：功能点（例如：用户注册、用户登录）
- 第三级：具体任务（例如：设计注册页面原型、开发注册接口、编写注册测试用例）

功能需求：【粘贴需求清单】
团队构成：【填入团队成员角色，例如：1 名产品、2 名前端、3 名后端、1 名测试】

同时输出：
- 任务总工作量估算（合计人天）
- 关键路径标识（哪些任务是不能延误的）
- 可并行执行的任务组

格式：三级任务表 + 关键路径说明
```

#### 审查提示词

```
请检查以下任务清单是否存在以下问题：
1. 是否有任务描述过于模糊（无法判断完成的标志）？
2. 是否有任务工作量估算明显不合理（过大或过小）？
3. 依赖关系是否有遗漏（某个任务实际上依赖另一个任务但未标注）？
4. 是否所有功能需求都能在任务清单中找到对应任务？

【粘贴任务清单】
原始需求：【粘贴需求清单】

逐项说明问题。
```

#### 修正提示词

```
根据审查意见，修改任务清单。补充模糊任务的完成标志，修正工作量估算，补充遗漏的依赖关系，确保每条需求都有对应任务。

原始任务清单：【粘贴原始清单】
审查意见：【粘贴审查结果】
```

**验收标准**：

- [ ] 每个任务是否都有明确的交付物（完成的标志）？
- [ ] 所有功能需求是否都能在任务清单中找到对应任务？
- [ ] 关键路径上的任务是否已被标识？

---

### 节点 4：风险识别专家

**职责**：基于任务清单和项目信息，识别项目执行中的潜在风险，并为每个风险提供应对措施。

**输入**：节点 3 的任务清单 + 已知约束条件

**输出**：风险清单（含风险描述、发生概率、影响程度、风险等级、应对措施）

#### 基础提示词

```
请根据以下项目任务清单，识别主要风险。

任务清单：【粘贴任务清单摘要】
已知约束：【填入截止日期、人力、技术限制等】

对每个风险输出：
- 风险描述（一句话）
- 发生概率（高/中/低）
- 影响程度（高/中/低）
- 风险等级（高/中/低，由概率和影响共同决定）
- 应对措施（具体操作，不是"密切关注"这类废话）
```

#### 进阶提示词

```
你是一位资深项目风险管理专家。请对以下项目进行系统性风险识别和应对规划。

项目概要：【粘贴项目背景、任务量和约束条件】

风险识别维度（请覆盖所有维度）：
1. 需求风险（需求变更、需求不清晰）
2. 技术风险（技术难点、第三方依赖）
3. 资源风险（人员变动、工作量低估）
4. 时间风险（节假日、外部审批周期）
5. 沟通风险（干系人意见不一致、信息传递失真）
6. 外部风险（监管变化、竞品动作）

对每个识别出的风险，提供：
- 风险触发条件（什么情况下这个风险会变成现实）
- 预防措施（在风险发生前能做什么）
- 应急预案（风险发生后的处理方案）

输出格式：风险登记表 + Top 3 高优先级风险的详细应对方案
```

#### 审查提示词

```
请检查以下风险清单是否存在以下问题：
1. 是否有风险的应对措施是"密切关注"或"及时沟通"这类空洞表达？
2. 是否遗漏了明显的常见项目风险类型？
3. 高风险项是否都有具体的应急预案？

【粘贴风险清单】

逐项说明问题。
```

#### 修正提示词

```
根据审查意见，修改风险清单。将空洞的应对措施替换为具体操作，补充遗漏的风险类型，为高风险项补充应急预案。

原始清单：【粘贴原始风险清单】
审查意见：【粘贴审查结果】
```

**验收标准**：

- [ ] 每个风险的应对措施是否是具体可操作的动作？
- [ ] 是否覆盖了需求、技术、资源、时间四类基本风险？
- [ ] 高风险项是否都有对应的应急预案？

---

### 节点 5：排期规划专家

**职责**：基于任务清单、依赖关系和资源情况，生成可视化的项目排期计划。

**输入**：节点 3 任务清单（含依赖关系、工作量）+ 节点 4 风险清单 + 可用人力和截止日期

**输出**：项目排期表（甘特图文字版）+ 里程碑清单

#### 基础提示词

```
请根据以下任务清单和资源信息，生成项目排期计划。

任务清单：【粘贴任务清单】
项目开始日期：【填入日期】
截止日期：【填入日期】
团队人力：【填入人数和角色】

请输出：
1. 以周为单位的甘特图文字版（每周列出进行中的任务）
2. 关键里程碑清单（3～5 个，每个注明日期和交付内容）
3. 缓冲时间说明（在哪里留了缓冲，留了多少）
```

#### 进阶提示词

```
你是一位资深项目排期专家，熟悉关键路径法（CPM）和资源平衡。

请根据以下信息，生成详细的项目排期方案：

任务清单（含工作量和依赖）：【粘贴清单】
团队资源：【填入每人角色和可用工作日】
项目开始日期：【填入日期】
硬性截止日期：【填入日期】
已知不可用时间段：【填入节假日或不可用时段】

排期要求：
1. 按关键路径优先排列任务
2. 避免同一人员同一天超过 8 小时工作量
3. 在第一个里程碑前保留 10% 缓冲时间
4. 高风险任务前一周设置检查点

输出格式：
- 里程碑清单（日期 + 交付内容 + 验收方式）
- 分周任务分配表（每周完成哪些任务，按阶段归类）
```

#### 审查提示词

```
请检查以下排期计划是否存在以下问题：
1. 是否有任务的前置依赖未完成就开始排期（违反依赖关系）？
2. 是否有某人或某角色在某周明显过载（超过合理工作量）？
3. 是否在关键节点前预留了检查和缓冲时间？
4. 排期总工期是否在截止日期以内？

【粘贴排期计划】
截止日期：【填入日期】

逐项说明问题。
```

#### 修正提示词

```
根据审查意见，调整排期计划。修正依赖违反问题，平衡人员工作量，在关键节点前补充缓冲时间，确保在截止日期以内完成。

原始排期：【粘贴原始排期】
审查意见：【粘贴审查结果】
```

**验收标准**：

- [ ] 排期是否在截止日期之前完成所有任务？
- [ ] 是否有明确的里程碑节点和验收说明？
- [ ] 关键路径上是否留有缓冲时间？

### 节点 6：资源评估专家

**职责**：评估项目所需的人力、工具和外部资源，识别资源缺口并提出补充方案。

**输入**：节点 3 任务清单 + 节点 5 排期计划

**输出**：资源需求报告（含已有资源、缺口资源、补充方案）

#### 基础提示词

```
请根据以下项目任务和排期，评估资源需求并识别资源缺口。

任务清单：【粘贴任务清单】
当前可用资源：【填入现有人力、工具清单】

请输出：
1. 总人力需求（各角色的人天汇总）
2. 工具/系统需求清单
3. 资源缺口（哪些资源不足）
4. 补充方案（外包/临时借调/推迟需求）
```

#### 进阶提示词

```
你是一位资深项目资源经理。请对以下项目做完整的资源评估，并给出资源优化建议。

项目信息：【粘贴任务清单和排期摘要】
现有资源：【填入人力和工具信息】
预算上限（可选）：【填入预算信息】

评估内容：
1. 人力资源分析（各角色需求量 vs 实际可用量，缺口说明）
2. 技术工具评估（项目需要哪些工具，是否已有授权）
3. 外部依赖资源（第三方 API、外包团队、采购周期）
4. 资源风险（哪个资源如果出现问题会卡住整个项目）
5. 优化方案（如何用最少的资源完成最核心的需求）

输出格式：资源需求表 + 缺口分析 + 优化建议
```

#### 审查提示词

```
请检查以下资源评估报告是否存在以下问题：
1. 是否有遗漏的资源类型（例如遗漏了测试环境或上线权限）？
2. 资源缺口的补充方案是否可行（是否符合实际情况）？
3. 是否识别了关键路径上的资源瓶颈？

【粘贴资源评估报告】
```

#### 修正提示词

```
根据审查意见，补充遗漏的资源类型，修正不可行的补充方案，标识关键路径上的资源瓶颈。

原始报告：【粘贴原始报告】
审查意见：【粘贴审查结果】
```

**验收标准**：

- [ ] 是否列出了所有角色的人力需求和缺口？
- [ ] 资源补充方案是否在项目时间范围内可执行？
- [ ] 是否识别了最关键的资源依赖？

---

### 节点 7：沟通计划专家

**职责**：设计项目的沟通机制、报告节奏和升级流程，确保干系人在正确的时间收到正确的信息。

**输入**：节点 2 干系人清单 + 节点 5 排期计划

**输出**：沟通计划（含会议节奏、报告格式、升级机制）

#### 基础提示词

```
请根据以下干系人清单和项目排期，设计项目沟通计划。

干系人清单：【粘贴干系人清单】
项目周期：【填入开始和结束日期】

请输出：
1. 定期会议安排（每种会议的频率、参与人、时长、议程模板）
2. 状态报告模板（谁写、给谁看、多久一次、包含哪些内容）
3. 升级机制（什么情况需要升级，升级给谁）
```

#### 进阶提示词

```
你是一位资深项目经理，擅长干系人沟通管理。请设计完整的项目沟通计划。

干系人清单（含权力/利益矩阵）：【粘贴干系人清单】
项目排期：【粘贴里程碑清单】
项目规模：【填入团队规模和项目周期】

沟通计划要素：
1. 信息需求矩阵（每个干系人需要收到什么信息，频率和格式）
2. 会议设计（区分决策会/同步会/汇报会，设计各自的议程结构）
3. 状态报告设计（交通灯模式：绿/黄/红，含触发条件）

5. 反馈收集机制（如何让团队成员反馈风险和阻碍）

输出：沟通计划文档（可直接在项目启动会上使用）
```

#### 审查提示词

```
请检查以下沟通计划是否存在以下问题：
1. 是否有重要干系人没有被纳入任何沟通机制？
2. 会议是否过于频繁（占用太多工作时间）或过于稀疏（信息滞后）？
3. 升级机制是否有明确的触发条件（不是"视情况而定"）？

【粘贴沟通计划】

```

```
原始沟通计划：【粘贴原始计划】
```

**验收标准**：

- [ ] 是否每个关键干系人都有对应的沟通安排？
- [ ] 升级机制是否有明确的触发条件（而不是"视情况"）？


## 常见错误

**错误 1：跳过需求澄清，直接排期**

表现：需求还没说清楚，就急着排时间表，导致计划做完了需求大幅变更。

修复：节点 1 的待确认项必须在排期前完成确认，或者在排期中明确标注"待确认后调整"。

**错误 2：风险应对措施写成"密切关注"**

表现：风险识别很详细，但应对措施是"密切监控""保持沟通"，发生风险时没人知道该怎么办。

修复：每个高风险项必须有"如果 X 发生，则立即执行 Y"格式的应急预案。

**错误 3：复盘放到项目结束后才开始设计**

表现：项目过程中没有收集数据，复盘时全靠回忆，复盘报告变成"感受总结"。

修复：在节点 8 确定"在项目执行中要记录哪些数据"，从项目第一天就开始收集。

## 延伸玩法

- **变体 1：敏捷版**：将任务拆解为 Sprint 周期，用节点 3 + 节点 5 生成两周迭代计划，而不是瀑布式全量排期。
- **变体 2：微型版**：对于周期不超过 2 周的小任务，只执行节点 1（需求澄清）+ 节点 3（任务拆解）+ 节点 4（风险识别），省略其余节点。
- **进阶组合**：与"工作流 15（领导汇报工作流）"结合，用本工作流的里程碑清单直接驱动下一次汇报材料生成。
