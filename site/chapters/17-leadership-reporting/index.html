    工作流 17：从零散工作材料到领导汇报稿的 AI 多专家工作流¶

    适用岗位：职能人员、项目负责人、部门助理、管理人员、汇报材料撰写者

    真实场景：领导让你明天汇报近期工作，你手上是一堆零散的材料——做过的事、遇到的问题、几个数据、一些未完成的项。你需要把它们整理成一份重点清楚、问题不甩锅、有价值的汇报稿，而不是流水账或空洞官话。

    最终目标：一份务实的领导汇报稿，包含：提炼后的重点成果、恰当表达的问题与风险、数据支撑、清晰的逻辑结构、分寸得当的措辞，以及对领导可能追问的预判准备。

    输入材料清单¶

    • 零散工作材料（做过的事、进展、数据、问题、未完成项）
    • 汇报对象（汇报给谁：直属领导 / 跨部门 / 高层）
    • 汇报场景（例会 / 专项汇报 / 述职 / 紧急汇报）
    • 汇报时长或篇幅要求（口头 5 分钟 / 书面一页）
    • 特别关切（领导近期关注什么，如有）

    工作流总览¶

    节点 1：原始材料整理专家
  ↓ 输出：分类后的工作材料清单
节点 2：汇报对象分析专家
  ↓ 输出：汇报对象关注点 + 表达基调
节点 3：重点成果提炼专家
  ↓ 输出：有价值的成果（而非事务罗列）
节点 4：问题与风险表达专家（招牌节点）
  ↓ 输出：得体表达的问题/风险/困难/建议
节点 5：数据支撑专家
  ↓ 输出：核对后的数据支撑点
节点 6：逻辑结构专家
  ↓ 输出：汇报逻辑框架
节点 7：措辞分寸专家
  ↓ 输出：措辞调整建议
节点 8：汇报稿生成专家
  ↓ 输出：完整汇报稿
节点 9：领导追问预判专家
  ↓ 输出：追问预判 + 应答要点
最终交付：领导汇报稿 + 应答准备

    专家节点详解¶

    每个节点包含六部分：节点定位、输入与输出、使用顺序、提示词包（A 快速 / B 专家 / C 自查 / D 返修）、交付给下游节点、人工验收清单。

    C 自查审稿版和 D 返修优化版是当前节点内部的工作模式，不是新的专家角色。

    核心原则：汇报是把"做了什么"提炼成"有什么价值"，把"遇到的问题"表达得实事求是又不推责。不编造成果，不夸大也不隐瞒问题。

    节点 1：原始材料整理专家¶

    1.1 节点定位¶

    将零散的工作材料分类整理，分清哪些是成果、哪些是进展、哪些是问题、哪些是数据。汇报前先把手里的"原材料"理清，才能提炼。

    1.2 输入与输出¶

    输入：零散工作材料 + 汇报场景

    输出：分类后的材料清单（成果 / 进行中 / 问题 / 数据 / 未完成）

    1.3 使用顺序¶

    1. 先用「快速生成版」分类整理。
    2. 材料杂乱或跨多个项目时，改用「专家增强版」。
    3. 用「自查审稿版」检查分类是否准确、是否遗漏。
    4. 有问题则用「返修优化版」修正。
    5. 对照 1.6 验收清单确认，通过后交给节点 3。

    1.4 提示词包¶

    A. 快速生成版¶
    你是一位工作材料整理员。请将以下零散材料分类。

工作材料：【粘贴所有零散材料】

按以下类别整理：
1. 已完成的成果
2. 进行中的工作
3. 遇到的问题
4. 相关数据
5. 未完成/待推进的事项
忠实分类，不夸大成果，不隐瞒问题。
    B. 专家增强版¶
    你是一位工作材料梳理专家。

任务：将零散工作材料分类整理，为汇报提炼打基础。

输入：
- 工作材料：【粘贴所有材料】
- 汇报场景：【填入例会/述职等】
- 汇报周期：【填入本周/本月/本季度】

处理步骤：
1. 按类别归集：已完成成果 / 进行中工作 / 遇到的问题 / 数据 / 未完成事项
2. 对每项标注与汇报周期的相关性（本期的 / 跨期延续的）
3. 标注每项的重要程度（核心工作 / 常规事务）
4. 标记信息不全的材料（如只有结论没有数据支撑）

输出格式：
| 类别 | 内容 | 重要程度 | 信息完整度 |

约束：忠实分类，不夸大成果也不淡化问题；不确定归类的标注存疑；不补充材料中没有的内容。

质量标准：材料分类清晰，核心工作和常规事务区分开，为后续提炼成果和表达问题做好准备。

常见失败情况：把常规事务当成核心成果；遗漏了问题类材料只留成果；把进行中的工作写成已完成。
    C. 自查审稿版¶

    此为当前节点的自查模式，不是新的专家角色。

    请检查以下材料分类是否存在以下问题：
1. 是否把进行中的工作误归为"已完成成果"？
2. 是否遗漏了问题类材料（只整理了成果）？
3. 核心工作和常规事务是否区分开？
4. 是否补充了材料中没有的内容？

【粘贴材料分类】

逐条说明问题。
    D. 返修优化版¶
    根据自查意见修正材料分类。将误判为已完成的进行中工作归位，补回遗漏的问题，区分核心与常规，删除凭空补充的内容。

原始分类：【粘贴原始分类】
自查意见：【粘贴自查结果】

    1.5 交付给下游节点¶

    将分类材料清单复制，交给节点 3（重点成果提炼专家）。汇报对象分析（节点 2）可并行进行。

    1.6 人工验收清单¶

    • [ ] 是否没有把进行中工作误写成已完成？
    • [ ] 是否保留了问题类材料？
    • [ ] 核心工作和常规事务是否区分？

    节点 2：汇报对象分析专家¶

    2.1 节点定位¶

    分析汇报对象的关注点和沟通偏好，确定汇报的基调和重点。同样的工作，向直属领导和向高层汇报，重点和措辞完全不同。

    2.2 输入与输出¶

    输入：汇报对象 + 汇报场景 + 特别关切（如有）

    输出：汇报对象关注点 + 表达基调 + 内容取舍建议

    2.3 使用顺序¶

    1. 先用「快速生成版」分析汇报对象。
    2. 汇报对象层级高或情况复杂时，改用「专家增强版」。
    3. 用「自查审稿版」检查关注点判断是否合理。
    4. 有问题则用「返修优化版」修正。
    5. 对照 2.6 验收清单确认，通过后交给节点 3 和节点 6。

    2.4 提示词包¶

    A. 快速生成版¶
    你是一位汇报沟通顾问。请分析汇报对象的关注点。

汇报对象：【填入直属领导/跨部门/高层】
汇报场景：【填入】
领导近期关注（如有）：【填入或留空】

输出：
1. 汇报对象最关心什么（结果/过程/数据/风险）
2. 表达基调建议（详细/精炼/正式/务实）
3. 哪些内容该重点讲，哪些可略过
    B. 专家增强版¶
    你是一位职场汇报沟通专家。

任务：分析汇报对象，确定汇报的重点取舍和表达基调。

输入：
- 汇报对象：【填入层级和角色】
- 汇报场景：【填入】
- 特别关切：【填入领导近期关注的事项，如有】

分析维度：
1. 关注重点（该层级通常关心结果还是过程、数据还是趋势、风险还是机会）
2. 信息颗粒度（要详细过程还是只要结论）
3. 表达基调（正式/务实/简洁）
4. 内容取舍（哪些必讲、哪些选讲、哪些不必讲）
5. 沟通禁忌（该对象反感什么，如冗长、报喜不报忧、甩锅）

输出格式：
- 关注点分析
- 表达基调建议
- 内容取舍清单（必讲/选讲/略过）
- 沟通注意事项

    节点 4：问题与风险表达专家（招牌节点）¶

    4.1 节点定位¶

    这是本工作流的招牌节点。 解决汇报中最微妙、最考验分寸的部分——怎么表达问题、风险、困难和建议。常见困境：

    困境 错误做法 本节点要解决的
    怎么写问题但不甩锅 把责任推给别人/客观条件 客观陈述问题，聚焦解决而非追责
    怎么写风险但不制造恐慌 危言耸听或轻描淡写 如实评估风险等级，给出应对
    怎么写困难但不显得推责 强调困难为完不成找借口 说明困难同时表明已采取的应对
    怎么写建议但不越权 替领导做决定或指手画脚 提供建议供参考，把决策权留给领导

    汇报里的问题表达，既不能粉饰太平（领导被蒙在鼓里），也不能甩锅诉苦（显得不担当）。

    4.2 输入与输出¶

    输入：节点 1 分类材料（问题/未完成部分）+ 节点 2 汇报对象

    输出：得体表达的问题/风险/困难/建议（每项含表达方式和分寸说明）

    4.3 使用顺序¶

    1. 先用「快速生成版」表达问题和风险。
    2. 问题敏感或涉及他人责任时，必须使用「专家增强版」。
    3. 用「自查审稿版」检查是否甩锅、是否制造恐慌、是否越权。
    4. 有问题则用「返修优化版」修正。
    5. 对照 4.6 验收清单确认，通过后交给节点 6。

四类表达原则：

1. 问题表达（不甩锅）
   - 客观陈述发生了什么，聚焦"现状和影响"而非"谁的错"
   - 涉及他人责任时，陈述事实即可，不主动归咎也不替人背锅
   - 表明自己已做或将做什么

2. 风险表达（不制造恐慌）
   - 如实评估风险等级（高/中/低），不夸大也不轻描淡写
   - 说明风险触发条件和潜在影响

   - 避免诉苦式表达

4. 建议表达（不越权）
   - 用"建议""供参考""是否可以考虑"等措辞

   - 不替领导拍板，不指手画脚

常见失败情况：把问题归咎于客观条件或他人（甩锅）；强调困难为完不成铺垫（推责）；建议用命令式口吻（越权）；风险危言耸听或刻意淡化。

    根据自查意见调整汇报结构。突出重点，把重要内容前置，补充明确的结尾诉求，调整篇幅分配。

原始结构：【粘贴原始结构】
自查意见：【粘贴自查结果】

    将汇报逻辑框架复制，交给节点 7（措辞分寸专家）。

    6.6 人工验收清单¶

    • [ ] 重点是否突出（而非平铺直叙）？
    • [ ] 重要内容是否在靠前位置？
    • [ ] 结尾是否有明确的下一步或诉求？

    节点 7：措辞分寸专家¶

    7.1 节点定位¶

    调整汇报的措辞分寸，让表达务实得体，既不卑微也不张扬，去除空话套话。措辞是汇报的"最后一公里"，分寸不当会让好内容打折扣。

    7.2 输入与输出¶

    输入：节点 6 逻辑框架 + 节点 4 问题表达 + 节点 2 表达基调

    输出：措辞调整建议（原表述 → 建议表述 + 理由）

    7.3 使用顺序¶

    1. 先用「快速生成版」调整措辞。
    2. 涉及敏感表达或正式场合时，改用「专家增强版」。
    3. 用「自查审稿版」检查是否有空话套话、分寸是否得当。
    4. 有问题则用「返修优化版」修正。
    5. 对照 7.6 验收清单确认，通过后交给节点 8。

    7.4 提示词包¶

    A. 快速生成版¶
    你是一位措辞顾问。请调整汇报措辞。

汇报内容要点：【粘贴节点6框架和节点4表达】
表达基调：【粘贴节点2】

调整原则：
1. 去除空话套话（赋能、抓手、闭环、提效等）
2. 务实具体，不夸大不卑微
3. 数字和事实说话，少用形容词

输出：| 原表述 | 建议表述 | 理由 |
    B. 专家增强版¶
    你是一位职场措辞分寸专家。

任务：调整汇报措辞，使表达务实、得体、有分寸。

输入：
- 汇报框架和要点：【粘贴节点6】
- 问题表达：【粘贴节点4】
- 表达基调：【粘贴节点2】

调整维度：
1. 去空话（删除"赋能、抓手、闭环、提效、深入推进"等空泛词，换成具体表述）
2. 控分寸（成果不夸大、不邀功；问题不卑微、不诉苦）
3. 重事实（用数字、事实、可验证的结果说话，减少主观形容词）
4. 调语气（符合表达基调：对高层简洁有力，对直属领导可适当详细）
5. 避雷区（避免绝对化表述、避免给自己挖坑的承诺）

输出格式：
| 原表述 | 建议表述 | 调整理由 |
+ 整体措辞基调说明

约束：措辞务实，不教粉饰或卖弄；保留内容的真实性，只调整表达方式；不堆砌华丽辞藻。

质量标准：汇报读起来务实可信，没有空话套话，分寸得当。

常见失败情况：用空泛词替换了具体内容；为显得"有水平"堆砌术语；成果夸大或问题表达过于卑微。
    C. 自查审稿版¶

    此为当前节点的自查模式，不是新的专家角色。

    请检查以下措辞调整是否存在以下问题：
1. 是否还残留空话套话（赋能、抓手、闭环等）？
2. 调整后是否用空泛词替换掉了具体内容？
3. 措辞分寸是否得当（成果不夸大、问题不卑微）？
4. 是否为显水平堆砌了不必要的术语？

【粘贴措辞调整】

逐条说明问题。
    D. 返修优化版¶
    根据自查意见修正措辞调整。清除残留空话，把被空泛化的内容改回具体表述，校正分寸，去除堆砌的术语。

原始措辞调整：【粘贴原始内容】
自查意见：【粘贴自查结果】

    将措辞调整建议复制，交给节点 8（汇报稿生成专家）。

    7.6 人工验收清单¶

    • [ ] 是否清除了空话套话？
    • [ ] 是否没有用空泛词替换具体内容？
    • [ ] 措辞分寸是否得当？

    节点 8：汇报稿生成专家¶

    8.1 节点定位¶

    整合前序所有产出，生成完整的汇报稿。汇报稿要可直接用于口头汇报或书面提交。

    8.2 输入与输出¶

    输入：节点 3 成果 + 节点 4 问题 + 节点 5 数据 + 节点 6 结构 + 节点 7 措辞

    输出：完整汇报稿

    8.3 使用顺序¶

    1. 先用「快速生成版」生成汇报稿。
    2. 正式汇报或书面提交时，改用「专家增强版」。
    3. 用「自查审稿版」检查汇报稿是否务实、是否符合结构和分寸。
    4. 有问题则用「返修优化版」修正。
    5. 对照 8.6 验收清单确认，通过后交给节点 9。

    8.4 提示词包¶

    A. 快速生成版¶
    你是一位汇报稿撰写员。请整合内容为汇报稿。

问题与风险：【粘贴节点4】

    B. 专家增强版¶
    你是一位专业汇报稿撰写专家。

- 逻辑结构：【粘贴节点6】
- 重点成果：【粘贴节点3】

    节点 9：领导追问预判专家¶

    9.1 节点定位¶

    预判领导可能追问的问题，准备应答要点。汇报常常不是讲完就结束，领导的追问才是关键时刻——有准备和没准备差别很大。

    9.2 输入与输出¶

    输入：节点 8 汇报稿 + 节点 4 问题风险 + 节点 2 汇报对象

    输出：追问预判清单（可能的追问 + 应答要点）

    9.3 使用顺序¶

    1. 先用「快速生成版」预判追问。
    2. 汇报涉及敏感问题或重大事项时，改用「专家增强版」。
    3. 用「自查审稿版」检查预判是否切中要害、应答是否可用。
    4. 有问题则用「返修优化版」补充。
    5. 对照 9.6 验收清单确认。

    9.4 提示词包¶

    A. 快速生成版¶
    你是一位汇报应答顾问。请预判领导可能的追问。

汇报稿：【粘贴节点8】

输出：
| 可能的追问 | 应答要点 |
重点预判：成果的真实性、问题的解决方案、数据的来源、下一步的可行性。
    B. 专家增强版¶
    你是一位汇报应答准备专家。

任务：预判领导可能的追问，准备应答要点。

输入：
- 汇报稿：【粘贴节点8】
- 问题与风险：【粘贴节点4】
- 汇报对象关注点：【粘贴节点2】

预判维度：
1. 成果追问（这个成果具体怎么实现的、价值怎么衡量、能否持续）
2. 数据追问（数据来源、口径、为什么是这个数）
3. 问题追问（问题根因、为什么会发生、怎么解决、什么时候解决）
4. 风险追问（风险有多大、如果发生怎么办）
5. 计划追问（下一步具体怎么做、需要什么资源、什么时候完成）
6. 责任追问（这事谁负责、为什么没早发现）

对每个预判追问准备应答要点：
- 应答方向（如实回答 / 需要补充确认 / 提供方案）
- 关键要点（应答时要说清楚的核心）
- 注意事项（这个问题回答时的分寸）

输出格式：
| 可能的追问 | 追问类型 | 应答要点 | 注意事项 |
+ 高概率追问标注（最可能被问到的 3 个）

约束：应答要点基于真实情况，不教编造或搪塞；对答不上的问题，建议如实说"需要会后核实"而非硬答。

质量标准：覆盖高概率追问，应答要点务实可用，帮助汇报者从容应对。

常见失败情况：只预判简单问题漏掉尖锐问题；应答要点教人搪塞；没有标注高概率追问。

1. 是否覆盖了尖锐问题（而非只预判好回答的）？
2. 应答要点是否教人搪塞或编造（应基于真实情况）？
3. 对可能答不上的问题，是否建议"会后核实"而非硬答？
4. 是否标注了高概率追问？

【粘贴追问预判】

    常见错误¶

    错误 1：把"做了很多事"直接当汇报内容

    表现：汇报变成流水账，列了一大堆做过的事，领导记不住也看不出价值。

    修复：执行节点 3，从"做了什么"提炼"产生了什么价值"，归并琐碎事务，突出真正的成果。

    错误 2：问题表达甩锅或诉苦

    表现：说问题时把责任推给客观条件或别的部门，或者强调困难为完不成铺垫，显得不担当。

    修复：执行节点 4（招牌节点），客观陈述问题聚焦解决，说困难时同时说应对，把握"如实但不推责"的分寸。

    错误 3：用空话套话掩盖内容空洞

    表现：通篇"赋能、抓手、闭环、深入推进"，听起来很热闹，实际没说清楚做了什么、有什么价值。

    修复：执行节点 7，清除空泛词，用数字、事实、具体结果说话。

    • [ ] 成果是否提炼出了价值，而非流水账罗列？
    • [ ] 问题表达是否客观得体，没有甩锅或诉苦？
    • [ ] 数据是否真实，没有编造或估算？
    • [ ] 是否清除了空话套话？
    • [ ] 是否准备了领导追问的应答要点？

    延伸玩法¶

    • 变体 1：书面报告版：将汇报稿扩展为正式书面报告，增加背景说明和详细数据附表，适合需要存档的正式汇报。
    • 变体 2：紧急汇报版：对突发问题的紧急汇报，重点走节点 4（问题表达）→ 节点 6（结构）→ 节点 9（追问预判），快速准备"问题-影响-应对-建议"四段式汇报。
    • 进阶组合：与"工作流 16（会议纪要工作流）"结合，把会议纪要的决策和成果作为汇报材料输入；与"工作流 18（数据分析报告工作流）"结合，用数据分析结论作为汇报的数据支撑。
